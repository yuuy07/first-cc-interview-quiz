# LLM Wiki Interview — Raw 层构建 Skill

本 Skill **只负责**向 **`raw/`** 写入与整理资料，**不**负责 `wiki/` 的实体页、索引或日志。与任何「从 raw 编译 wiki」的流程的衔接方式只有一个：**共享的 `raw/` 目录与其中文件**。

## 目标

1. **结构化**：在 `raw/` 中产出可追溯、可引用的两层 Markdown：**基础层（basic）**与**整合层（blog）**。
2. **不重复**：基础解释只锚定权威来源；博客层负责架构视野与概念整合，**不与 basic 重复堆叠定义**。
3. **可验证**：主张尽量带来源链接；博客类**忠实于原文**（见第二阶段）。

## 目录约定（相对知识库根或用户指定的 `raw` 根）

```
raw/
├── _research.md              # 唯一检索总账（raw 包级）：按「一级 # + 二级 ##」组织；见下文「检索笔记标题层级」
├── basic/                    # 第一阶段：主题拆分后的「基础概念」树
│   └── {子主题}/
│       └── *.md              # 单文件结构见「面试向长文模版（basic）」
├── blog/                     # 第二阶段：优质博客/长文（整合视角）
│   └── *.md
└── assets/                   # 图片等（可按来源或日期分子目录）
    └── ...
```

若用户已有 `raw/`，在现有结构下**增量扩展**，不删除用户已有文件；新内容使用清晰文件名避免覆盖。

### 篇目文件命名（`basic/`、`blog/` 下 `.md`）

- 使用**稳定、可读的语义化文件名**（主题或来源 slug，与正文标题大致对应即可）。  
- **禁止**在文件名中使用：「补充」「修订」「备忘」「草稿」等临时性字样，以及**日期串**（如 `2026-04-19-…`）作为文件名的主要区分手段。  
- 同一主题反复迭代时：**就地修订**同一文件并更新 frontmatter 的 `updated`，**不要**另存为「第二版 / 带日期」的新文件名，以免目录膨胀与重复阅读成本。

### 阅读范围（省 token，默认遵守）

raw 目录会频繁增改，**禁止**把「通读 `raw/basic/**` 与 `raw/blog/**` 下每一篇」作为默认前置步骤。

1. **总是先读** `raw/_research.md`：映射表、各 `##` 下索引与备注，弄清已有子目录与篇目路径。  
2. **再只读**本轮会改动的文件：用户点名的路径，或映射表 / `_research.md` 中明确列出的篇目；必要时用**按文件名或关键词搜索**定位单文件，**不要**无差别批量打开全部 Markdown。  
3. **迭代前后对照**：在 `_research.md` 的映射表「备注」或对应 `###` 下用**一两句话**写清本轮「计划改哪些路径、改完与否」即可；不必在总账里复制大段正文 diff。  
4. 若用户未给出具体文件、且映射表不足以判断：先向用户确认**目标路径**，再读文件；仍优于全量通读。

### 内部互链规范（`raw/` 内 basic / blog / `_research.md`）

凡引用 **同一 `raw/` 资料包内**的另一篇 Markdown（含 `_research.md`、各 `basic/**` 篇目、`blog/*.md`），须使用 **标准 Markdown 链接**，以便 Obsidian / 编辑器等可点击跳转：

- **格式**：`[显示文本](相对路径)`；显示文本建议用**短标题或文件名**，避免裸 URL 式文件名堆叠。  
- **相对路径**：以**当前文件所在目录**为基准。示例：`basic/eval-guardrails-cost/foo.md` 引用同包内其它篇目用 `../model-gateway/某篇.md` 或 `../../blog/某篇.md`（按实际层级计算）；`blog/bar.md` 引用 basic 用 `../basic/eval-guardrails-cost/foo.md`；**总账** `raw/_research.md` 引用 basic 用 `basic/子目录/文件名.md`，引用 blog 用 `blog/文件名.md`；`basic/.../foo.md` 若需指向总账则用 `../../_research.md`。  
- **篇目 ↔ 总账**：`basic/**` 与 `blog/**` 下的笔记**应积极互链**（概念相邻、上下游、同一子域对照），**不强制**在正文里加入「返回 `_research.md`」类链接；总账侧通过映射表维护入口即可（Obsidian 反向链接可辅助查看引用关系）。  
- **禁止**：仅以反引号包裹 `` `某篇.md` ``、`raw/.../某篇.md` 作为**唯一**互链形式（不可点击）；代码标识、外链、YAML 值中的路径仍可按需使用反引号。  
- **`_research.md` 映射表**（总账内维护）：「篇目」列中 basic 条目写为 `[标题或文件名](basic/子目录/文件名.md)`；`blog/` 条目写为 `[…](blog/文件名.md)`（路径均相对 **`raw/` 根**下的 `_research.md`）。

### 检索笔记：唯一 `_research.md`（硬规则）

1. **唯一落点**  
   - 所有调研轮次、所有子主题（含 LLMOps、LiteLLM、源码深化等）的**检索笔记内容**只写入 **`raw/_research.md`**（**raw 包总账**，与 `basic/`、`blog/` **同级**）。  
   - **禁止**用「无限追加顶层 `##` + 仅日期/轮次标签」作为长期结构；见下文 **「检索笔记标题层级」**。

2. **禁止新建其它 `_research*.md`**  
   - **不得**创建除 `_research.md` 以外的任何匹配 `_research*.md` 的文件（例如禁止 `_research-llmops.md`、`_research-llmops-litellm.md`、`_research-topic.md`）。  
   - 深化、子域、源码细节等：只在 **`_research.md` 中**写入对应 **二级 `##` 小节**（及下属 `###`），和/或 **增量修订** `raw/basic/**` 下对应篇目；**不**用新开检索文件来区分主题。

3. **与历史文件并存**  
   - 若仓库中已存在历史命名的 `_research-*.md`（旧规范遗留），**本 Skill 执行时仍不得再新增**任何新的 `_research*.md`；新内容一律写入 **`_research.md`**。是否将旧文件合并进 `_research.md`、是否删除旧文件，由用户决定，本 Skill **默认不自动删除**用户已有文件。

4. **主题边界与越界确认**  
   - 若本轮素材与当前 raw 包所覆盖的范围明显不一致（且无法合理视为对已有 basic/检索章节的**延续或归类内扩展**），视为**越界**：**先暂停落盘**，说明归类冲突并**征得用户确认**后再写入 `raw/`（检索部分仍只写入 **`_research.md` 中正确的 `##`/`###`**）。

5. **`raw/basic/{子主题}/` 与检索笔记的分工**  
   - `{子主题}/` 目录仅用于组织 **basic 长文**；**检索索引与调研日志**只存在于 **`_research.md`**，不因目录变深而再建任何 `_research*.md`。

### 检索笔记标题层级（`_research.md` 结构规范）

目的：`_research.md` 位于 **`raw/` 根**，既是**调研记录**，也是 **整个 raw 资料包（basic + blog）的目录与索引入口**：读一遍标题即可建立「子目录 ↔ 子主题 ↔ 已覆盖关键词/链接」的心智模型，**避免**随轮次增加导致顶层 `##` 无限变长。

**标题级别（最多用到三级）**

| 级别 | Markdown | 含义 | 约束 |
| --- | --- | --- | --- |
| 一级 | `#` | **领域包 / raw 资料包总名**（如当前 `raw` 包覆盖的大主题：`LLMOps`、`RAG 工程` 等，由用户或仓库约定） | **全文件建议只有一个一级标题**（或 frontmatter + 一个 `#` 总标题） |
| 二级 | `##` | **通常**与 `raw/basic/` 下直接子目录 **一一对应**（见下「目录映射」） | 例：子目录为 `model-gateway/` 则二级标题用 `## model-gateway`（可加括号注「LiteLLM 等」）。**同一业务子域的所有检索追加 → 只在该 `##` 下用 `###` 组织**，不得再新开顶层仅作「日期分区」的 `##`。**例外**：允许 **一个**不对应任何子目录的 `##`（如「领域级：子域关键词与面经」），用于跨子域索引，**内容只写在 `_research.md`**，**不**要求另建 `overview/` 或独立 basic 篇目（由用户选择是否保留该节）。 |
| 三级 | `###` | 该子域下的 **子话题、关键词块、索引小节** | 标题用**语义化短语**（如 `### 网关：降级与重试要点`、`### 关键词与面经`）。**禁止**用「以日期或日期加临时说明词为主导」的 `###` 标题当目录结构（例如用日期、或「日期 + 临时说明」起头的标题）；时间信息只放在 frontmatter 的 `updated` 或正文短句，不占用标题命名空间 |

**目录映射（强制维护）**

- **`raw/basic/<子目录>/`** 必须在 `_research.md` 中有对应 **`## <标题>`**（与子目录对齐）。  
  - **推荐**：`##` 标题与 **文件夹名** 一致或可直接辨认（例：文件夹 `model-gateway/` → `## model-gateway` 或 `## 模型网关（model-gateway）`）。  
  - **特例**：`basic/` 下尚无子目录（或尚未落任何篇目）时，不强行建与空目录对应的 `##`。  
  - **跨子域首轮/总览**：可仅在 `_research.md` 中设 **额外** `##`（无对应文件夹），**不**强制创建 `overview/` 目录或重复篇目；映射表中用一行注明「无子目录」。  
- 在 **一级 `#` 段落后、各 `##` 之前**，维护 **「`basic/` 子目录 ↔ `_research.md` 二级标题 ↔ 目录内篇目（文件名）」** 对照表（可随每次迭代更新 `updated`）：  
  - 列示例：`子目录`、`research 中 ## 标题`、`篇目（*.md）`、`备注`。  
  - 这样 **只读 `_research.md` 即可遍历 `raw/basic/**` 已有内容**，并知道每块调研笔记落在哪个子目录下。

**与 basic 长文的关系**

- 每个 `##` 下建议包含：**子关键词 / 面试追问 / 索引链接**（可拆成多个语义化 `###`）；**新增的 basic、blog 路径** 写在 **对应子目录的 `##`（或其 `###`）** 内，并同步更新映射表。  
- **历史遗留**：若仍存在旧规范的「顶层按日期堆叠的 `##`」，迁移方式为：将内容并入 **对应子目录的 `##` + 语义化 `###`** 下；是否执行迁移由用户确认，Skill **仅在用户明确要求后代为整理**。

### 检索笔记内容结构（`_research.md`）

目的：在 `raw/` 长期扩容时，**检索关键词、面试关注点、索引链接**有稳定落点，且**结构随目录生长而非随日期堆叠**。

- 文首可选 **YAML frontmatter**：`title`（如「Raw 检索总账」）、`type: research`、`created`、`updated`。  
- 正文：**一个一级标题** + **映射表** + **若干 `##`**（多数与 `basic/` 子目录对齐；可选一个跨子域 `##` 无对应目录）+ 各 `##` 下用 **语义化 `###`** 组织子关键词表、面试高频追问、索引级链接、产出路径等。  
- **时间信息**：迭代时间以 frontmatter `updated` 或映射表「备注」、正文必要处短句记录即可；**不用日期式 `###` 标题**充当目录结构。  
- **目录映射表**（见上）建议放在 **`#` 与第一个业务 `##` 之间** 或 **文首 frontmatter 之后**，便于一眼导航。

## 第一阶段：关键词拆分 + 面试方向 + Basic 骨架

### 输入

- 用户主题（可替换为任意领域）。
- 可选：岗位层级、面试深度、时间预算。
- 可选：**用户指定文章 / Blog**（见第二阶段 **「用户供稿」**）：**公开 URL**、**本地文件路径**（仓库内相对路径优先；工作区外路径由执行环境可读则抽取正文）、**聊天内粘贴全文**；可与「仅主题检索」组合执行。**本地稿与链接稿同一套落盘与总账/basic 增量要求。**

### 动作

1. **互联网检索（可多轮）**
   - 从主题拆分 **3～10 个** 子关键词或子方向。
   - 检索「主题 + 面试」「子关键词 + interview」「面经」等，整理 **高频追问方向**（抓代表性即可）。
   - 将检索结论写入 **`raw/_research.md`**：关键词表、面试关注点、索引级链接（正文仍以权威文档为主）；**写入与 `basic/` 子目录对应的 `##` 小节**，同一子域内用 **语义化 `### 子话题名`** 追加，**不覆盖**已有段落；并更新映射表。**禁止**新建其它 `_research*.md`。

2. **生成 `raw/basic/` 目录树**
   - 每个子主题对应子文件夹或合并为一组文件（避免过度碎片化）。
   - 每篇 basic 解决：**是什么、核心术语、与相邻概念边界、与面试追问相关的要点**。

3. **内容来源约束（第一阶段）**
   - **优先**：官方文档、官方博客、GitHub README/docs、RFC、规范。
   - **避免**：把第二阶段的个人/团队博客整合文写进 basic，以免与 blog 层重复。
   - **写法**：技术说明可核对到上述来源；无官方依据时标注「推断/待核实」。

### 面试向长文模版（basic 专用）

`raw/basic/` 下每篇笔记应采用**同一套章节骨架**，便于口述备考与后续任意工具消费。以下为**抽象后的写法要求**（不绑定某一题库或外部文件路径）：

**1. YAML frontmatter（建议字段）**

- `title`：与文件名或主题一致。
- `tags`：主题、子主题、领域标签。
- `type: basic`
- `sources`：本页引用的权威 URL 列表（数组或分行）。
- `created` / `updated`：日期。

**2. `## 精炼回答`**

- 用 **数段完整中文**，把该子主题答成一道「口述题」：先给**结论与主干**，再点出**关键术语**与**边界**。
- 语气：**可直接照着说**的压缩版，避免只列名词不串逻辑。

**3. （可选）配图**

- 若需图，放在 `raw/assets/`，正文用 **相对路径** 引用。

**4. `## 扩展分析`**

- 按**面试展开**来写，而不是维基式定义罗列。推荐层次：
  - **破题**：一句话说清问题在考什么（系统边界、环节、trade-off）。
  - **主线展开**：按「先易后难」或「常见追问顺序」分段；每段可含**口述示例句**（用引号标出即可）。
  - **易混对比**：成对概念（如 Precision/Recall、两类 Relevance）**对照讲清**。
  - **追问防御**：「若面试官追问 X」时如何答到**本质**而非背公式（公式点到为止，重在适用场景与局限）。
  - **串起来**：各指标/模块之间的**依赖与级联**（例如检索噪声如何传导到生成）。
- 可穿插简短分层总结（如「分三层：检索 / 生成 / 端到端」），便于收尾。

**5. `## 实战落地`**

- 从「**项目里怎么做**」落笔：数据怎么建、离线怎么批跑、用什么 judge、示例代码或 Prompt 片段、与业务指标如何对齐。
- 代码或 Prompt **注明**来源仓库、文档章节或自建说明，避免无来源长代码。

**6. 文末「参考链接」**

- 列出本页用到的官方/权威链接，与 frontmatter 的 `sources` 可合并或互补。

> **说明**：上述骨架对应「精炼口述 → 深度展开 → 工程落地」三层；第二阶段 **blog** 的编译类篇目须遵守下文 **「Blog：frontmatter 摘要 + 正文结构」**。允许在 **YAML frontmatter** 中放置结构化摘要（要点/思路/面试借鉴），**禁止**在正文里、在 **`## 全文译文` / `## 全文正文` 章节标题之下**用「大段摘要体」替代按序完整呈现（来源合规等 **一句** 声明仍允许）。

## 第二阶段：博客与长文（整合层）

### 定位

- **目的**：从更高视角串联 basic 中的概念（架构、trade-off、生产实践），**扩写整合视角**而非复述 basic。
- **选题**：优先多概念整合、架构决策、踩坑与演进；排除与某篇 basic 同题的纯入门科普（除非用户要求补缺）。

### 检索策略（选题来源）

- **中文社区**：检索词以 **中文** 为主（如「主题 + 实践」「子关键词 + 架构」「面经」「工程落地」等），覆盖专栏、团队博客、技术社区长文；优先信息密度高、可核对出处的篇目。
- **英文社区**：用 **英文** 检索高信息密度博客/官方工程文/深度专栏，与中文侧形成互补（架构词、产品名、英文固定术语组合检索）。
- **用户供稿（指定文章）**：用户给出的 **URL、本地文档、粘贴全文** 均视为 **高优先级选题**，**流程与产出标准一致**（`raw/blog/` + `_research.md` + `basic` + 剩余配额内主动检索）；与主动检索 **共用**「每轮 blog ≤5」计数。
- **组合**：同一轮选题中宜 **中英兼有**（视主题可用度调整），避免单一语料偏见；所有篇目仍须写入 **`raw/_research.md`** 的索引与映射表。

### 用户供稿（网络 URL / 本地文档 / 粘贴全文）

**三种形态等价**：用户提供的待收录材料可以是 **线上文章链接**、**本地已有文档**（如知识库内的 `.md`、`.docx`、`.pdf` 等——先抽取可读正文再处理），或 **聊天/编辑器内粘贴的全文**。不要求必须有 URL；**只要用户明确指定为「供稿」**，即按本节与网络稿相同的质量标准落入 `raw/blog/`，并同步 **`_research.md` 检索索引**与 **`basic/**` 增量**，再在剩余配额内继续丰富 blog。

当用户**显式提供**一篇或多篇待收录文章时，在遵守 **Blog frontmatter**、**全文译文/全文正文 + 核心内容提取**、图片本地化等规则前提下，按下列顺序执行（可与第一阶段/迭代穿插，但落盘顺序建议如下）。

1. **解析与归类**  
   - **取材**：URL 则抓取正文；**本地路径**则读取该文件（若格式非 Markdown，先抽取正文再进入后续步骤，版式以可读为准）；粘贴稿则直接使用用户给出的文本。  
   - 确认语言：中文 → `## 全文正文`；外文 → `## 全文译文`。  
   - 将文章归入 **`raw/basic/<子主题>/` 所对应的 `_research.md` 二级 `##`**；若跨子域，在总账中用 **语义化 `###`** 标明「用户供稿」块，并在映射表备注中写清归类理由。  
   - 若单轮用户供稿篇数 **超过 5**：与用户确认 **优先队列** 或拆分为多轮执行；**禁止**单轮突破 `blog` 上限。

2. **落入 `raw/blog/`**  
   - 每篇独立 `.md`，语义化文件名；`source` / `author` 等据实或据文首推断填写。  
   - **`url` 字段**：有公开链接则填真实 URL；**纯本地或粘贴稿**无 URL 时，使用 `url: ""`（或省略，与仓库内其它编译文一致即可），并在 **`note`** 中写明来源类型，例如「用户本地文档」「用户粘贴稿」；若原始文件在**本知识库仓库内**，建议在 `note` 或单独字段 `local_source` 中写 **相对仓库根的路径**（便于协作与回溯），**避免**在共享仓库中硬编码个人本机绝对路径（敏感时可只写文件名 + 一句说明）。  
   - **外文**：完整译为简体中文写入 `## 全文译文`。  
   - **中文**：正文保持原文写入 `## 全文正文`。  
   - 补齐 **`extracted_points` / `key_ideas` / `interview_hooks`** 与 `## 核心内容提取`。  
   - **与本地原件的关系**：供稿落盘为 `raw/blog/` 下的**规范编译篇**；**不强制**删除或改写用户原始本地文件；若用户希望「单一日志入口」，可在 `_research.md` 的供稿索引中同时链到 `blog/新篇.md` 与（若适用）**库内**原始路径。

3. **写入「basic 检索」侧（总账 + basic 篇目）**  
   - **`raw/_research.md`**：在对应 `##` 下追加 **`###` 用户供稿索引**（或并入已有索引小节），包含：文章标题、本地 `[blog/…](blog/…)` 链接、**从文中抽取的子关键词 / 面试追问 / 与 basic 的对照点**；更新映射表「篇目」列。  
   - **`raw/basic/**`**：**不**把整篇 blog 定义搬进 basic；对该文涉及的概念，在已有 basic 中 **增量修订**（「扩展分析 / 实战落地」增补可口述要点，并注明观点来自哪篇 `blog/…`）；若出现 **无对应 basic 的缺口**，按「面试向长文模版」**新建** basic 篇目，权威定义仍以官方文档为主，用户文仅作 **工程视角 / 架构叙述** 的补充并标注来源。

4. **继续丰富 blog（主动检索）**  
   - 在完成用户稿落盘与总账/basic 增量后，用 **本轮剩余配额**（`5 − 用户稿相关新增/大改篇数`）按 **「检索策略」** 主动检索并新增或修订 `raw/blog/`，使用户包在「指定深度」与「社区广度」上兼顾。  
   - 若剩余配额为 0，在回复中说明「本轮额度已满，主动检索延至下一轮」。

### 数量要求（硬约束：上限 5）

- **每轮 Skill 执行**（含「首轮建 raw」与「同一 raw 根上迭代深化」）：对 `raw/blog/` 的工作量为 **「新增编译篇目数 + 本轮做实质性大改的既有 `blog/*.md` 篇数」合计 ≤ 5**。  
  - **首轮建包**：在不超过 5 篇的前提下，按信息密度与去重原则尽量覆盖多视角；**不**再以「至少 3 篇」作为未完成判据。  
  - **迭代**：在不超过 5 篇的前提下追加或修订 blog；若用户明确只要 basic、本轮不写 blog，须在回复中说明并跳过本上限计数。  
- **去重**：与已有 `raw/blog/*.md` 同题或正文高度重叠的，**不**计入有效新增；宁少勿滥。

### 动作

1. 若用户本轮提供了指定文章：先按 **「用户供稿」** 完成解析、归类、`blog/` 落盘、`_research.md` 与 `basic` 增量；再基于 **关键词表 + 面试关注点** 与 **剩余 blog 配额**，按「检索策略」检索博客/专栏/工程文。若无用户供稿，则直接基于关键词与面试关注点检索。
2. **忠实性（编译类：frontmatter 摘要 + 硬性两阶段正文）**
   - 中文来源：在 `raw/blog/` 中每来源独立 `.md`；**正文保持中文原文**，不做无意义的「再翻译」。
   - 外文：**须先完成「全文级」转写，再允许提炼**——见下文「Blog 正文结构：全文译文 + 核心提取」；**禁止**仅用摘要、提纲、段首段尾节选或「改写浓缩版」冒充整篇 blog。
   - frontmatter：**除** `source`、`author`、`url`、`fetched` 或 `translated`、`language`（如 `zh-Hans`）等惯例字段外，**必须**包含下文 **「Blog frontmatter：文首 YAML 摘要块」** 所列摘要字段（简体中文撰写）；正文可按需加编译说明（**编译说明不得替代全文译文**）。
3. **图片**
   - 下载至 `raw/assets/`（如 `raw/assets/blog/{slug}/`），Markdown 内 **相对路径** 引用。
   - 命名避免冲突（如 `{来源-slug}-{序号}-{简述}.png`，**避免**仅靠日期串区分）。
   - 无法下载时，在文内「资源说明」记录 URL 与原因。
4. **去重**
   - 若正文大量重复某篇 basic 的定义句，**跳过**或只保留整合独有段落，并说明取舍。

### Blog frontmatter：文首 YAML 摘要块

每篇 `raw/blog/*.md`（编译类）在 **正文任何 Markdown 标题之前**，须有完整 **YAML frontmatter**（`---` … `---`）。在惯例字段之后，**必须**增加 **简体中文** 的结构化摘要（可与 `note` 合并或分字段，但信息不可缺）：

| 字段（推荐键名） | 内容要求 |
| --- | --- |
| `extracted_points` | **要点提取**：条目或短段落，覆盖原文关键论断与事实 |
| `key_ideas` | **重要思路**：论证链、架构取舍、方法论与设计直觉 |
| `interview_hooks` | **面试可借鉴**：可改述为口述的追问方向、对比维度、踩坑与防御性答法 |

允许使用 YAML 多行字符串（`|`）或列表 `-`；键名可微调（如 `digest` 单块内分三段），但三类信息 **必须都能在 frontmatter 中一眼找到**。**禁止**用 frontmatter 摘要替代下文「全文译文/全文正文」的完整呈现。

### Blog 正文结构：全文译文 + 核心提取

frontmatter 之后，正文**必须**拆为两大块，**顺序固定**，且第二块不得穿插进第一块的叙述流：

1. **`## 全文译文`**（或 **`## 全文正文`**；中文原文照搬时**必须用后者或等价名**以标示「非译制改写」）  
   - **外文**：对原文 **从头到尾** **完整翻译**为简体中文；段落、小节标题、列表、表格、代码块顺序与原文一致；**不跳段、不合并论点、不用「大意如下」替代原文句子**。  
   - **中文来源**：**保持原文中文**，**禁止**为凑格式而全文「再翻译」或机械同义改写；可修正明显错别字与排版。仍须使用独立章节名（如 `## 全文正文`）以便与第 2 块区分。  
   - **代码块**：须与来源一致；若抓取/HTML 导致代码被压成单行，仅允许做 **空白与换行** 的可读化排版，**禁止**删改标识符、逻辑与字符串字面量。链接与专有名词可在译文句中保留英文。  
   - **站点页脚**（评论、登录引导、无关推荐列表等）**不**收录；**正文内**的作者相关推荐、原文「Related posts」若属于文章组成部分则须译出。

2. **`## 核心内容提取`**（或 **`## 要点提炼`**）  
   - 仅在本节做 **面试向** 提纲、对照表、高频追问——**必须在全文译文/全文正文之后另起一章**；**禁止**把本节要点打散混入第一块以替代完整翻译或完整原文呈现。  
   - 与 frontmatter 的 `extracted_points` / `key_ideas` / `interview_hooks` **允许呼应但避免大段逐字重复**：本节可侧重表格、追问清单、与 basic 的对照；frontmatter 侧重「打开文件即读」的总览。

> **自检**：若删除「核心内容提取」整章后，剩余正文仍不足以覆盖原文的信息点与顺序（外文），或中文原文被不当删改，则视为 **未完成**，须返工。

## 迭代更新（同一 raw 根）

当用户**在已有内容基础上**继续加深某一子主题（例如已存在 `raw/basic/model-gateway/LiteLLM统一网关与多供应商路由.md`，现要求「再调研 LiteLLM」「深化网关实践」），执行下列流程；执行前**默认只精读**本 Skill 本节与 **`raw/_research.md`**，并按上文「阅读范围」打开**点名篇目**，避免通读全库。如果点名篇目中链接了其他篇目，可以一并读取并调整

### 触发语（示例）

- 「细化 / 深化 / 追加 + 子主题或技术名」
- 「在已有 LLMOps raw 上再加 LiteLLM」
- 「更新 basic 并加 blog」
- 「收录这篇 / 这些 URL 到 blog，并补检索与 basic」
- 「用户供稿：{链接 / 本地文件路径 / 粘贴正文}，再帮我丰富相关 blog」

### 必做步骤

1. **定位现有篇目（不默认全量阅读）**  
   - **先读** `raw/_research.md` 的映射表与各 `##` 下索引。  
   - 再用关键词搜索或用户给出的路径，**仅打开**将涉及的 `.md`；**禁止**把「列出并读完 raw 下全部 Markdown」作为固定步骤。  
   - 列出：**将复用的路径**、**需扩写的篇目**、**需新建篇目的缺口**。

2. **更新检索笔记**  
   - 仅在 **`raw/_research.md`** 中、**对应子目录的 `##` 小节下**追加：使用 **语义化 `###` 小标题**（如 `### 网关：本轮检索要点`），写清本轮新增关键词、面试追问、计划改动的 basic/blog 清单；并**更新** **「子目录 ↔ `##` ↔ 篇目」映射表**与 `updated`。若尚无对应 `##`，先新增该 **`##`（与子目录同名或可辨认）** 再写内容。**不得**新建任何其它 `_research*.md`。

3. **更新或新增 basic**  
   - 若已有对应文件：在**不改变原意**前提下**增量修订**（更新 frontmatter 的 `updated`、增补「扩展分析 / 实战落地」中与本轮相关的要点）。  
   - 若无对应文件：按「面试向长文模版」**新建**，并放入语义清晰的 `{子主题}/` 目录。

4. **追加或修订 blog（整合层）**  
   - 若本轮含 **用户供稿**：先完成 **「用户供稿」** 小节中的 blog 落盘与 `_research.md` / `basic` 增量，再在 **剩余配额** 内按「检索策略」继续丰富 blog。  
   - 否则：针对本轮**新增信息增量**，按「检索策略」检索并写入或修订 `raw/blog/`，遵守 **「数量要求：上限 5」**（新增 + 本轮大改合计 ≤ 5）。  
   - blog 须覆盖本轮**区别于既有 basic 的整合视角**（架构、踩坑、对比），避免与已存在的 `raw/blog/*.md` 同题重复。

5. **收束输出**  
   - 在回复中明确：**本轮更新的是哪个 `##`（及可选 `###`）**、**映射表是否有变更**、**basic 变更列表**、**blog 新增与修订列表**（并注明是否触发「每轮 blog ≤5」计数）。
